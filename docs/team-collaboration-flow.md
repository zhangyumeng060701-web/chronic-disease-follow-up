## 二、核心协作链：前端C ↔ 1号后端

这是最重要的配合，每天都会发生。

### 工作节奏

```
1号后端                      前端C
────────                    ──────
周一：写随访CRUD接口         
      Postman自测通过
                             周二：拿到接口地址
                                  开始调真实API
周三：修复C调不通的问题       
     ← 群里说"字段名不对"
     → "应该用patientId不是patient_id"
                             周三：继续对接
                                  调通随访列表

周五：写预警接口              
                             周五：做预警页面Mock
                                  等接口就绪下周对接
```

### 铁律

- 1号每个接口写完 → **在群里发 Postman 截图**，写清楚 URL 和参数
- 前端C调不通 → **群里 @1号，贴报错截图**，不要自己猜
- 字段名争议 → **查 swagger-api.md**，以文档为准
- 1号不要等C说"可以了"再写下一个接口（前端Mock数据可以先做页面）

## 三、1号后端 ↔ 5号测试

```
1号写完患者CRUD接口
  → 群里通知5号"患者接口就绪，URL在swagger-api.md"
  → 5号写 PatientControllerTest
  → 测试不过 → 群里@1号贴失败日志
  → 1号修 → 5号重跑 → 通过
```

5号不需要等前端对接完才测。后端接口 **Postman 能通就可以开始写测试**。

## 五、4号维护平台 ↔ 前端C

- 4号定稿 `ui-spec.md` 后 → 发群里给C看
- C做页面时配色/字体/组件风格以 ui-spec.md 为准
- C提 PR 时 → @4号作为 Reviewer
- 4号检查：有没有 console.log？有没有行内样式？字段名对不对？

---

## 六、6号部署 ↔ 全员

```
6号写好 docker-compose.yml
  → 发群里
  → 每人执行: docker-compose up -d
  → 验证: curl http://localhost:8080/api/health
  → 在群里发截图打卡
  → 有人启动失败 → 6号排查
```

6号每周五检查一次 GitHub 仓库：有没有人直接在 develop 提交？有没有 .env 泄露？

## 七、组长 ↔ 全员

```
每天早上：
  打开 GitHub → Pull requests → 看有没有新的PR没人Review
  打开飞书看板 → 看谁的任务逾期了 → 群里@那个人

每天下午：
  有人接口争议 → 你拍板
  有人被阻塞 → 你协调

每周五：
  汇总进度 → 发到群里
  准备下周任务
```

## 八、配合规则速查

| 谁 | 找谁 | 干什么 |
|---|---|---|
| 前端C | 1号后端 | 接口URL、字段名疑问、接口报错 |
| 1号后端 | 前端C | 通知接口就绪、确认C需要的字段 |
| 5号测试 | 1号后端 | 通知接口就绪可以开始测、Bug反馈 |
| 1号后端 | 5号测试 | 修完Bug通知回归测试 |
| 4号 | 3号 | 调AI拆解接口、格式确认 |
| 4号 | 前端C | Review代码、UI规范检查 |
| 6号 | 全员 | Docker报错排查、Git违规纠正 |
| 全员 | 组长 | 被阻塞、接口争议、权限问题 |
| 组长 | 全员 | 进度催办、PR未Review提醒 |

## 十、绝对禁止

- ❌ 接口调不通不找对方，自己闷头猜
- ❌ 改了接口字段名不通知对方（前端或后端单方面改名）
- ❌ 不等 Review 就自己 Merge PR
- ❌ 在群里只发"报错了"不贴截图
- ❌ 被阻塞了不吭声，等组长来问
