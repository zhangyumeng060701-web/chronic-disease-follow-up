# 测试项目面试复盘

## 1. 项目测试目标

本项目是一个基层慢病随访管理系统，涉及患者档案、随访记录、风险预警、统计看板和系统管理。测试目标不是只验证页面能点，而是保证医疗类系统中的核心数据、敏感信息和业务规则可信。

本阶段我负责测试与安全方向，重点目标包括：

- 搭建前后端自动化测试基础。
- 验证患者管理基础服务逻辑。
- 验证敏感数据脱敏规则。
- 梳理慢病异常预警规则。
- 识别认证、权限、依赖和敏感配置风险。

## 2. 测试策略和测试层级

我采用了分层测试策略：

- 单元测试：前端脱敏工具函数、后端 Service 逻辑。
- Controller 测试：使用 MockMvc 验证健康检查接口的统一响应格式。
- 文档与规则审查：检查 `t_alert_rule` 表和慢病异常规则是否一致。
- 安全审计：使用 `npm audit` 检查前端依赖风险。

这样做的原因是当前项目还处于早期阶段，随访、预警、统计等模块尚未全部实现，所以优先建立可以快速运行、后续能接入 CI 的测试基线。

## 4. 发现的典型问题

### 问题 1：后端测试无法运行

现象：执行 `mvn test` 时提示 `mvn` 命令不存在，`where.exe java` 也找不到 Java。

定位：这是环境问题，不是测试代码失败。当前机器缺少 JDK 和 Maven，或者没有配置 PATH。

处理：明确记录为阻塞项，后续在安装 JDK 11 和 Maven 后执行 `mvn test`。

### 问题 2：敏感信息脱敏不能只依赖前端

现象：当前患者接口直接返回 Entity，尚未在后端做脱敏。

风险：如果只靠前端组件隐藏，接口响应里仍然有明文，用户可通过浏览器网络面板看到敏感数据。

建议：后端应在 Service 层或 VO 转换层统一脱敏，前端组件只作为展示兜底。

### 问题 3：依赖审计发现漏洞

现象：`npm audit` 报告 ECharts、Vite/esbuild/Vitest 依赖链存在漏洞。

风险：包括 XSS 和开发服务器相关风险。

建议：不要直接在业务分支执行 `npm audit fix --force`，应单独开依赖升级分支，升级后执行自动化测试和页面回归。

### 问题 4：认证链路存在实现风险

现象：JWT 过滤器只把用户信息放进 request attribute，没有设置 Spring Security Authentication。

风险：受保护接口可能仍然被 Spring Security 判定为未认证。

建议：后续补充鉴权测试，并修复过滤器认证上下文。

## 5. 如何定位和修复

我的定位流程是：

1. 先读标准文档，明确必须交付什么。
2. 再读项目结构和已有代码，判断哪些模块已经实现。
3. 对已实现模块先写最小可运行测试。
4. 测试失败时区分环境失败、测试代码失败、业务代码失败。
5. 对安全问题先记录风险和修复建议，不随意做破坏性升级。

本阶段做过的修复包括：

- 将脱敏逻辑从组件中抽离为纯函数，方便单元测试。
- 使用 Unicode 转义修正测试样例，避免 Windows 编码导致的“乱码对乱码”假通过。
- 后端测试使用 MockMvc standalone 和 Mockito，减少对数据库和完整 Spring 上下文的依赖。

## 6. 如何保证回归测试

当前可重复执行的回归命令：

```bash
cd frontend-target
npm.cmd test
```

后端在环境具备后执行：

```bash
cd backend
mvn test
```

后续 CI/CD 应在 PR 阶段自动执行：

- `mvn test`
- `npm test`
- `npm run build`
- 依赖安全审计

## 8. 面试官可能追问与回答思路

### 你为什么先测脱敏函数，而不是直接测页面？

因为脱敏是核心规则，抽成纯函数后可以快速、稳定地单元测试，不依赖浏览器和 UI。页面组件只负责调用工具函数，后续再补组件测试或 E2E 测试。

### 为什么后端测试不用真实数据库？

当前阶段目标是先验证业务逻辑和测试框架，使用 Mockito mock Mapper 可以避免数据库环境影响测试稳定性。数据库相关行为后续可以用集成测试或 Testcontainers 补充。

### 发现依赖漏洞为什么不直接修？

因为 `npm audit fix --force` 会引入破坏性升级，可能导致 Vite、Vitest、ECharts 版本变化，影响构建和页面行为。正确做法是单独开依赖升级分支，升级后完整回归。

### 医疗系统为什么强调后端脱敏？

前端脱敏只能控制展示，不能保护接口响应。真正的安全边界应该在后端，非管理员查询时后端就应该返回脱敏数据，避免通过网络面板或接口调用拿到明文。

### 你怎么判断测试是否真的有效？

我会看测试输入是否对应真实业务规则，断言是否验证关键结果，失败时是否能暴露真实问题。本次还修复了乱码测试数据，避免出现“测试通过但业务无效”的情况。
